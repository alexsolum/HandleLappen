<objective>
Wire Supabase Realtime subscriptions so that changes made on one device appear on all other devices viewing the same list or home screen within 3 seconds — without a page refresh.

Purpose: This is the LIST-06 requirement and the defining UX of a real-time collaborative shopping list. The implementation is intentionally minimal: `postgres_changes` events → `invalidateQueries`. No manual cache splicing, no complex merge logic — TanStack Query re-fetches and the UI updates.
Output: Two `$effect` blocks in the two key pages, and a two-context Playwright test proving sub-3-second cross-device sync.
</objective>

From 02-RESEARCH.md — Realtime subscription pattern (Pattern 4):
```typescript
// In a Svelte 5 component — use $effect for lifecycle management
import { useQueryClient } from '@tanstack/svelte-query'
import { onDestroy } from 'svelte'

// Setup inside $effect:
const channel = supabase
  .channel(`list-items-${listId}`)          // unique channel name per list
  .on(
    'postgres_changes',
    {
      event: '*',                             // covers INSERT, UPDATE, DELETE
      schema: 'public',
      table: 'list_items',
      filter: `list_id=eq.${listId}`,         // server-side filter — only events for this list
    },
    () => {
      queryClient.invalidateQueries({ queryKey: ['items', listId] })
    }
  )
  .subscribe()

// Cleanup in onDestroy:
onDestroy(() => supabase.removeChannel(channel))
```

CRITICAL PREREQUISITE (from 02-RESEARCH.md Pitfall 1):
- The `list_items` and `lists` tables MUST be in the `supabase_realtime` publication
- This was done in the migration from plan 02-01
- If events are not firing: check Supabase Dashboard > Database > Publications > supabase_realtime
- Silent failure mode: `.subscribe()` resolves without error but callback never fires

Anti-pattern to AVOID (from 02-RESEARCH.md Pitfall 4):
- NEVER set up Realtime channels in `+page.server.ts` or `+layout.server.ts`
- WebSocket connections require a browser — must be in $effect or onMount only

import { useQueryClient } from '@tanstack/svelte-query'
import { onDestroy } from 'svelte'
import { itemsQueryKey } from '$lib/queries/items'

const queryClient = useQueryClient()

// Realtime: subscribe to list_items changes for this specific list
// The filter is server-side — only events for data.listId reach this client
const itemsChannel = data.supabase
  .channel(`list-items-${data.listId}`)
  .on(
    'postgres_changes',
    {
      event: '*',
      schema: 'public',
      table: 'list_items',
      filter: `list_id=eq.${data.listId}`,
    },
    () => {
      queryClient.invalidateQueries({ queryKey: itemsQueryKey(data.listId) })
    }
  )
  .subscribe()

onDestroy(() => {
  data.supabase.removeChannel(itemsChannel)
})
```

Note: `useQueryClient()` works here because the component is a descendant of the `QueryClientProvider` in the protected layout. Do NOT call `new QueryClient()` here.

import { useQueryClient } from '@tanstack/svelte-query'
import { onDestroy } from 'svelte'

const queryClient = useQueryClient()

// Realtime: subscribe to lists changes so the home screen updates when
// another device creates or deletes a list
const listsChannel = data.supabase
  .channel('household-lists')
  .on(
    'postgres_changes',
    {
      event: '*',
      schema: 'public',
      table: 'lists',
    },
    () => {
      queryClient.invalidateQueries({ queryKey: ['lists'] })
    }
  )
  .subscribe()

onDestroy(() => {
  data.supabase.removeChannel(listsChannel)
})
```

Note: The `lists` subscription does not need a filter by household_id because Supabase RLS (WALRUS) enforces household isolation server-side before events are delivered. The client will only receive events for lists it has RLS SELECT access to.

```typescript
import { test, expect } from '@playwright/test'
import { createClient } from '@supabase/supabase-js'
import { createHouseholdUser, createTestUser, deleteTestUser } from './helpers/auth'
import { createTestList } from './helpers/lists'

const SUPABASE_URL = process.env.PUBLIC_SUPABASE_URL ?? 'http://127.0.0.1:54321'
const SERVICE_ROLE_KEY = process.env.SUPABASE_SERVICE_ROLE_KEY ?? ''

test.describe('realtime', () => {
  test('realtime sync — item added on device A appears on device B within 3 seconds', async ({ browser: pw }) => {
    // Set up two users in the same household
    const { user: userA, household } = await createHouseholdUser('realtime-a@test.example', 'password123')
    // User B joins the same household
    const admin = createClient(SUPABASE_URL, SERVICE_ROLE_KEY, {
      auth: { autoRefreshToken: false, persistSession: false },
    })
    const { data: userBAuthData } = await admin.auth.admin.createUser({
      email: 'realtime-b@test.example',
      password: 'password123',
      email_confirm: true,
    })
    const userB = userBAuthData!.user
    await admin.from('profiles').insert({
      id: userB.id,
      household_id: household.id,
      display_name: 'Person B',
    })

    const list = await createTestList(household.id, 'Delt liste')

    // Create two browser contexts (two devices)
    const contextA = await pw.newContext()
    const contextB = await pw.newContext()
    const pageA = await contextA.newPage()
    const pageB = await contextB.newPage()

    try {
      // Log in both devices
      for (const [page, email] of [[pageA, 'realtime-a@test.example'], [pageB, 'realtime-b@test.example']] as const) {
        await page.goto('/logg-inn')
        await page.fill('[type=email]', email)
        await page.fill('[type=password]', 'password123')
        await page.click('[type=submit]')
        await page.waitForURL('/')
      }

      // Both devices navigate to the shared list
      await pageA.goto(`/lister/${list.id}`)
      await pageB.goto(`/lister/${list.id}`)

      // Device A adds an item
      await pageA.fill('input[placeholder="Legg til vare…"]', 'Appelsinjuice')
      await pageA.keyboard.press('Enter')

      // Device B should see the item within 3 seconds (without refresh)
      await expect(pageB.locator('text=Appelsinjuice')).toBeVisible({ timeout: 3000 })
    } finally {
      await contextA.close()
      await contextB.close()
      await deleteTestUser(userA.id)
      await deleteTestUser(userB.id)
    }
  })
})
```

Note on test environment: This test requires the local Supabase stack to be running (Docker Desktop + `npx supabase start`) and the `supabase_realtime` publication to include `list_items`. If the local stack is not running, the test will fail at login. This is expected and documented in the Phase 1 summary — Docker must be running for full integration tests.
