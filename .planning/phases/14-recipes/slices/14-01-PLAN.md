# Phase 14-01: Recipe Backend Foundation

## Goal
Implement the database schema, RLS policies, and Supabase Storage bucket required for the recipe system.

## Plan
1.  **Database Migration**:
    *   Create `recipes` table:
        *   `id` (uuid, PK, default `gen_random_uuid()`)
        *   `household_id` (uuid, FK `households.id`, NOT NULL)
        *   `name` (text, NOT NULL)
        *   `description` (text, NULL)
        *   `image_url` (text, NULL - stores the storage path, e.g., `recipes/uuid.webp`)
        *   `created_at` (timestamptz, default `now()`)
    *   Create `recipe_ingredients` table:
        *   `id` (uuid, PK, default `gen_random_uuid()`)
        *   `recipe_id` (uuid, FK `recipes.id`, ON DELETE CASCADE, NOT NULL)
        *   `name` (text, NOT NULL - normalized item name)
        *   `position` (integer, NOT NULL, default 0)
    *   Create RLS policies:
        *   `recipes`: Select/Insert/Update/Delete where `household_id = auth.my_household_id()`.
        *   `recipe_ingredients`: Select/Insert/Update/Delete where `recipe_id` belongs to a recipe in the user's household.

2.  **Supabase Storage**:
    *   Create `recipes` bucket (public).
    *   Apply RLS policies:
        *   INSERT: Authenticated users can upload to `recipes/{household_id}/*`.
        *   SELECT: Public access or authenticated users can read `recipes/{household_id}/*`.
        *   UPDATE/DELETE: Authenticated users can modify `recipes/{household_id}/*`.

3.  **Type Generation**:
    *   Run `npx supabase gen types typescript --local > src/lib/types/database.ts`.

## Testing Strategy
*   **Manual**: Use `supabase/migrations` to apply schema. Verify tables exist.
*   **Playwright**: Create `tests/recipes.spec.ts` (Wave 0) to verify database operations via helper functions (create/delete recipe).
*   **Storage**: Verify bucket creation and policy application via Supabase dashboard or CLI.
